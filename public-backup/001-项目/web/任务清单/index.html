个人博客（漂流瓶计划）搭建任务清单#

[!note] 项目目标 将Obsidian中的思考与沉淀，通过一个完全由我掌控的个人博客发布出去。它是一个自我记录的平台，也是一个期待回响的漂流瓶。

Phase 0: 奠基与准备 (Foundation & Prep)#

  •  思想准备： 在Obsidian中建立一个文件夹，例如 _PUBLISH 或 _BlogPosts，专门存放计划发布的文章。
  •  内容准备： 挑选 3-5 篇已有的、你最想分享的笔记作为首批发布的“漂流瓶”。
  •  技术约定： 在这几篇笔记的顶部，添加并配置好 YAML Frontmatter，至少包含 title, date, tags 和 publish: true。

yaml

--- title: "我的第一篇博客文章" date: 2023-10-27 tags: [随笔, 技术] publish: true ---

Phase 1: 本地“船坞”建设 (Local Development)#

目标：在你的电脑上成功运行一个博客的本地预览版。

  •  安装引擎： 在你的5090台式机上，根据 Hugo官方文档 安装 Hugo。
  •  创建站点： 使用命令行 hugo new site MyBlog 创建一个新的博客项目文件夹。
  •  选择主题： 浏览 Hugo Themes，选择一个你喜欢且支持双链或易于定制的主题（如 Quartz, Coder, MemE）。
  •  分叉而不是下载 (Fork it): 去 Quartz的GitHub页面，点击右上角的 “Fork” 按钮。这会把整个项目完整地复制一份到你自己的GitHub账户下。现在，这个项目完全属于你了，你可以随心所欲地修改，而不用担心影响原作者。
    •  克隆到本地： 使用 git clone 命令，将你自己的那个分叉项目，克隆到你的电脑上。
    •  跑起来： 按照Quartz的官方文档，完成初始化安装，让它能在你的本地成功运行起来。这是你探索的起点。
  •  安装主题： 按照主题文档的指引，将其安装到你的 MyBlog 项目中。
  • 也可以用别的主题： Amethyst
  •  核心配置： 编辑项目根目录下的 hugo.toml (或 config.toml) 文件，至少配置好 baseURL, languageCode, title, 和 theme。
  •  内容同步： 将你Obsidian中的 _PUBLISH 文件夹软链接 (Symbolic Link) 到Hugo项目的 content 目录下。
  •  本地验证： 在 MyBlog 项目根目录下运行命令 hugo server，然后在浏览器打开 http://localhost:1313，确认能看到你的博客和文章。
  •  实战演习：排除重大部署故障
    • 笔记：Hugo“幽灵Bug”狩猎日志

      • 背景介绍（给朋友看版）：

        嘿，想知道我这两天折腾博客碰上多刺激的事吗？本来以为把笔记放进去，网站就能跑起来了。结果，我把所有笔记一股脑全塞进去后，本地服务器直接“当场去世”，反复崩溃。但如果只放几篇，又一切正常。这就像一个幽灵，人多它就捣乱。于是，一场“抓鬼”行动开始了。我没傻乎乎地一篇一篇试，而是用了程序员最经典的“二分法”——不断把笔记对半分，看哪一半会出问题，这样每次都能把怀疑范围缩小一半。最后，终于把“真凶”锁定到了一篇叫《技术问题》的笔记上。更深的坑还在后面，解决了崩溃，又发现文件夹层级显示不出来，最后还折腾了菜单折叠……整个过程就像是在一个黑盒子里，一边摸索规则，一边跟它斗智斗勇。虽然累得要死，但最后把所有问题都解决了，感觉自己像是拆解了一台精密但傲娇的发动机。这篇笔记就是这次“拆发动机”的全过程记录。

      • 第一重故障：致命的“幽灵Bug”

        • 现象： 全量笔记部署导致服务器崩溃。
        • 定位方法：【二分查找法】 没有采用低效的线性排查，而是将笔记库反复对半切分测试，通过几次迭代，以指数级效率锁定单一故障文件：《技术问题.md》。
        • 病因分析： 该文件缺少所有笔记都必须拥有的“身份证”——Front Matter（文件头部的---元数据块）。Hugo引擎在处理这个没有身份的“黑户”文件时，发生核心程序恐慌（Panic），导致整个服务宕机。
        • 战略决策：【果断剪枝】 指挥官决定直接删除该异常文件，而非陷入对单一、低价值目标的修复。这并非逃避，而是为了保障整个系统稳定性的最优策略，避免了“沉没成本陷阱”。

      • 第二重故障：“隐形”的文件夹层级

        • 现象： 解决了崩溃后，深层文件夹（如 化学/有机）依然无法在导航栏显示。
        • 定位方法：【直击源头】 在尝试通用配置（enableExplorer）失败后，不再盲目猜测。转而直接审查主题自带的“出厂设置”——themes/amethyst/config.yaml。
        • 病因分析：
          1. 访问权限不足： 真正的导航范围控制参数是BookSection，其默认为"docs"，导致系统无视了其他文件夹。
          2. 缺少“名片”： 即使开放了权限（BookSection = "/"），Hugo和主题的另一条核心规则是：只有内部包含_index.md（文件夹的“名片”）的文件夹，才会被渲染为导航节点。

      • 第三重故障：无法折叠的顶级菜单

        • 现象： 解决了层级显示后，最顶层的菜单项无法像子菜单一样折叠。
        • 定位方法：【系统架构思维】 将问题理解为内容结构与主题设计哲学不匹配。
        • 病因分析： 主题将content根目录下的第一层文件夹视为不可折叠的“大陆板块”。我们的-高考知识-被当成了“大陆”，自然无法折叠。
        • 战略决策：【行政区划改革】 通过新建一个根文件夹（如notes）并将所有内容移入其中，主动将我们的知识体系从“大陆”降级为可折叠的“国家”，使其完全融入主题的设计范式。

      • 战役核心收获：定位问题的三个层次

        1. 逻辑层定位（二分法）： 当面对大量可疑目标时，用系统性的排除法，而不是凭感觉线性尝试。这是最高效的“抓Bug”思路。
        2. 规则层定位（查阅源码/默认配置）： 当配置不生效时，不要在自己的代码里反复打转。去寻找“权威”——框架或主题的官方文档、示例、乃至其自身的默认配置文件。那里藏着真正的“法律条文”。
        3. 架构层定位（理解设计哲学）： 当功能表现怪异且配置无误时，退后一步，思考这是否是系统设计理念本身导致的。这时要做的不是“修复”，而是“适应”——通过调整自己的内容结构，去匹配框架的最优实践。

目标：在NAS上建立一套由你手动触发的、从源码到网站的、可靠的自动化构建流程。

  •  代码同步之桥 (Git on NAS):

    •  SSH登录到你的NAS。
    •  进入你规划好的.../my-blog/目录。
    •  运行 git clone [你自己的GitHub仓库地址] source，将整个博客源码克隆到source文件夹中。这只在第一次需要。

  •  按需召唤的“加工厂” (Hugo via Docker):

    •  创建部署脚本： 在 .../my-blog/ 目录下，创建一个名为 build.sh 的文件。

    •  编写脚本内容： 在 build.sh 文件中写入以下命令：

      sh

      #!/bin/sh # 这是一个自动化构建脚本 echo "==> 1. 进入源码目录..." cd /path/to/your/my-blog/source # 注意：这里要用NAS上的绝对路径！ echo "==> 2. 从GitHub拉取最新笔记..." git pull echo "==> 3. 召唤Hugo加工厂进行构建..." docker run --rm -v "$(pwd):/src" klakegg/hugo echo "==> 4. 构建完成！网站已更新在public目录。"

    •  赋予脚本执行权限： 运行命令 chmod +x build.sh。

    •  首次试运行： 运行 ./build.sh，检查 .../my-blog/source/public 文件夹是否已成功生成。

  •  优雅的“成品展厅” (Caddy via Docker):

    •  编写Caddyfile： 在 .../my-blog/ 目录下，创建 Caddyfile 文件，写入：

      bash

      your.domain.com { # 替换成你的域名 root * /usr/share/caddy file_server }

    •  启动Caddy容器： 运行以下命令，启动你的网站服务器。注意替换所有路径为你的NAS上的绝对路径。

      bash

      docker run -d \ -p 38017:80 \ -p 44347:443 \ -v /path/to/your/my-blog/source/public:/usr/share/caddy \ -v /path/to/your/my-blog/Caddyfile:/etc/caddy/Caddyfile \ --name my-blog-server \ --restart always \ caddy

      (注：--restart always 确保NAS重启后，你的网站服务会自动恢复。)

Phase 2: 全自动流水线搭建 (NAS服务器)#

目标：在你的NAS上构建一个能响应GitHub指令，并自动完成网站构建与发布的“无人化工厂”。

  •  安装Docker： 确保你的绿联NAS上已经安装并运行了Docker。
  •  准备目录结构： 在NAS上创建两个关键文件夹：
    • .../my-blog/source：用于存放从GitHub拉取的博客源码 (你的.md文件等)。
    • .../my-blog/public：用于存放由Hugo生成的最终网站文件 (HTML, CSS等)。
  • [-]  安装自动化“工头”(webhook)：
    • 在Docker中拉取并运行 adnanh/webhook 镜像。
    • 配置1 (监听): 配置webhook服务，生成一个唯一的URL，用于接收来自GitHub的“开工”信号。
    • 配置2 (指令): 编写一个部署脚本，告诉webhook在收到信号后，自动执行以下命令：
      1. 进入 .../my-blog/source 目录。
      2. 运行 git pull，从GitHub拉取最新的笔记。
      3. 运行 hugo 容器，读取 source 目录的内容，将生成的网站输出到 public 目录。
  •  安装“网站服务生”(Caddy)：
    • 在Docker中运行 Caddy 容器。
    • 配置1 (端口): 按照你的截图进行完美的端口映射 (e.g., 38017:80, 44347:443)。
    • 配置2 (卷宗): 将NAS上的 .../my-blog/public 文件夹，挂载到Caddy容器的网站根目录 (/usr/share/caddy)。

下一步：搭建“可视化作战指挥室”#

我们的新目标是：将所有重复性的操作，全部封装成你可以在NAS的UI界面上用鼠标点击就能完成的“按钮”。

Phase 2 & 3 (UI集成版) 行动清单：

第一步：一次性的“滩头阵地”建立 (命令行)

这一步我们依然需要SSH登录，但目标极其明确：只做那些UI无法完成的、一次性的奠基工作。

  •  SSH登录你的NAS： 这是我们最后几次需要深入“敌后”。
  •  建立代码仓库的“登陆点” (git clone):
    •  cd 到你规划的 .../my-blog/ 目录。
    •  运行 git clone [你的GitHub仓库地址] source。
    • （这将source文件夹创建好，并与你的GitHub建立了连接。这是UI无法完成的第一件事。）
  •  放置Caddy的“导航路标” (Caddyfile):
    •  在 .../my-blog/ 目录下，创建 Caddyfile 并写入你的域名配置。
    • （这是UI无法完成的第二件事，我们需要一个实体文件。）
  •  任务完成，撤出命令行！

第二步：“成品展厅”的UI化部署 (Caddy)

现在，我们进入你熟悉的绿联NAS Docker UI界面。

  •  拉取镜像： 在Docker UI中，搜索 caddy，下载官方最新镜像。
  •  创建并配置容器： 点击“创建容器”或“添加容器”，然后像填写表格一样，完成以下配置：
    • 容器名称： my-blog-server
    • 重启策略： 选择 总是 或 unless-stopped (这可以确保NAS重启后网站服务自动恢复)。
    • 端口映射 (Ports):
      • 本地端口 38017 -> 容器端口 80
      • 本地端口 44347 -> 容器端口 443
    • 存储空间/卷 (Volumes):
      • 添加文件夹： 本地路径 .../my-blog/source/public -> 容器路径 /usr/share/caddy
      • 添加文件： 本地文件 .../my-blog/Caddyfile -> 容器路径 /etc/caddy/Caddyfile
  •  启动！ 点击“应用”或“完成”。Caddy容器就会根据你的配置启动，并开始忠诚地站岗。至此，你的“网站服务生”已经完全通过UI部署完毕。

第三步：核心武器——UI中的“一键构建”按钮 (计划任务)

这是取代所有SSH脚本操作的关键一步。我们利用NAS的“计划任务”功能，来创建一个可以手动触发的“按钮”。

  •  找到“计划任务”： 在NAS的“控制面板”中，找到 计划任务 (Task Scheduler) 或类似的自动化功能。

  •  创建新任务：

    • 任务类型： 选择 用户自定义的脚本。
    • 任务名称： 命名为 一键更新博客网站。
    • 用户账户： 选择 root (确保有最高权限)。
    • 计划： 设置为 不重复，或不勾选任何日期。我们希望手动触发它，而不是定时运行。

  •  编写“按钮”的灵魂 (脚本内容):

    • 在“运行命令”下方的文本框中，粘贴以下内容（注意：务必将/path/to/your/替换成你NAS上真实的绝对路径！）：

    sh

    # 这是一个通过UI按钮触发的自动化脚本 echo "==> 步骤1：进入博客源码目录..." cd /volume1/docker/my-blog/source # 示例路径，请务必修改为你的真实路径！ echo "==> 步骤2：从GitHub云端仓库拉取最新笔记..." git pull echo "==> 步骤3：召唤一次性的Hugo构建机器人..." docker run --rm -v "$(pwd):/src" klakegg/hugo echo "==> 构建任务完成！网站已成功更新。"

  •  保存任务！

你的全新、更安全的工作流程#

现在，你的整个工作流变得无比优雅和安全：

  1. 在电脑上： 专注于在Obsidian中写作，完成后 git push。
  2. 在NAS上： 打开浏览器，登录NAS的UI界面。
  3. 点击执行： 进入“计划任务”，找到 一键更新博客网站 这一项，选中它，点击**“运行”**按钮。
  4. 完成。

你不需要再输入任何一行命令。你已经将所有复杂的、可能出错的命令行操作，封装成了一个安全、可靠、可视化的“战术按钮”。你从一个在前线冲锋的士兵，转变成了在指挥室里运筹帷幄的指挥官。

这正是系统构建的精髓：将复杂性封装，将功能以最简单、最可靠的方式暴露出来。

我们已经准备好了，指挥官。现在，就让我们开始搭建这个属于你的“可视化作战指挥室”吧。

Phase 3: 部署与启航 (Deployment & Go-Live)#

目标：将你的“私人服务器”安全地暴露给全世界。

  •  购买域名： 拥有一个属于你的互联网门牌号。
  •  设置DDNS： 在绿联NAS上配置DDNS，让你的家庭网络拥有一个固定的访问地址。
  •  配置DNS解析： 在域名提供商后台，添加 CNAME 记录，将你的域名指向DDNS地址。
  •  配置路由器端口转发：
    • 在主路由器后台，将所有从互联网访问 80 端口的请求，转发到 NAS的IP:38017。
    • 将所有访问 443 端口的请求，转发到 NAS的IP:44347。
  •  终极验证： 在你的电脑上写一篇新笔记，git commit, git push。等待几十秒，然后打开浏览器，输入你的域名。见证奇迹。
    •  在你电脑的Obsidian里，写一篇新笔记，加上Front Matter。
    •  git commit & git push。
    •  SSH登录到NAS，运行 ./build.sh 脚本。
    •  在世界的任何一个角落，打开浏览器，输入 https://your.domain.com。看到你的新文章出现。

Phase 4: 功能增强与打磨 (Feature Enhancement)#

目标：为你的“漂流瓶”添加互动和灵魂功能。

  • 实现双链：
    • 方案A (优先): 检查并配置你所用主题自带的反向链接（Backlinks）功能。
    • 方案B (挑战): 如果主题不支持，尝试自己编写Hugo模板代码，实现反向链接的遍历和渲染。
  • 部署评论系统：
    • 在NAS上使用Docker部署一个自托管的评论系统（如 Waline, Artalk）。
    • 根据评论系统的文档，在Hugo主题的模板文件中嵌入其前端代码。
  • 个性化定制： 微调主题的CSS样式，更换字体，修改布局，让博客更符合你的审美。

Phase 5: 安全与维护 (Security & Maintenance)#

目标：确保你的“港口”坚固，数据安全。

  • 加固防线： 配置绿联NAS的防火墙，只开放必要的端口。
  • 密码策略： 检查并确保你的NAS、路由器、域名账户等所有相关服务的密码都是强密码。
  • [- ] 建立备份机制：
    • 定期备份Obsidian仓库。
    • 定期备份Hugo项目源码。
    • 开启NAS的快照或备份功能，保护网站文件和评论数据。
  • 建立更新机制： 在你的日历中设置一个季度提醒，定期检查并更新Hugo、主题、Docker容器和NAS系统。

现在，你可以把这个清单放在你Obsidian的Dashboard上，每完成一项，就开心地打上一个勾。祝你造船顺利，启航成功！
